## Plan: Undo/Redo 系统

**你的猜想完全正确**：核心就是两个栈（undo stack / redo stack）+ Command 对象。每个 Command 有 `execute()` 和 `undo()` 两个方法。EditHistory.js 已有这个骨架，只是还没有具体的 Command 子类和接入点。

---

### 架构设计

**Command 模式**：每个用户操作对应一个 Command 对象，记录"做了什么"和"如何反做"。

标签/字符串解码区域的偏移量恢复用**快照（snapshot）**方案——操作前序列化当前状态，undo 时直接还原，比逆向计算偏移更简单可靠（标签数量通常很少，内存可接受）。

**需要实现的 Command 类型**：

| 操作 | Command | undo 策略 |
|------|---------|----------|
| 替换字节（replace 模式） | `ReplaceCommand` | 写回旧字节 |
| 插入字节（insert 模式/粘贴） | `InsertCommand` | 删除插入内容 + 恢复 tags/decode 快照 |
| 删除字节（Backspace/剪切） | `DeleteCommand` | 重新插回已保存的字节 + 恢复快照 |
| 标签创建 | `TagCreateCommand` | `deleteTag(id)` |
| 标签编辑 | `TagUpdateCommand` | `updateTag(id, oldData)` |
| 标签删除 | `TagDeleteCommand` | 用原 id 重建标签（需新增 `createTagWithId()`） |
| 添加字符串解码区域 | `StringDecodeAddCommand` | `removeRegion(regionId)` |
| 调整文件大小 | `ResizeCommand` | resizeFile 回原大小 + 恢复 tags/decode 快照（截断时还需还原字节） |
