## 一、项目概述

HexEditor 是一个基于 Web 技术的十六进制编辑器，使用 Vue 3 + Vite 构建。项目采用现代前端架构，目标是提供类似桌面级十六进制编辑器的功能，同时具备 Web 应用的跨平台优势。

### 技术栈

| 模块 | 技术 | 版本 |
|------|------|------|
| 前端框架 | Vue 3 | 3.5.13 |
| 构建工具 | Vite | 6.3.3 |
| 状态管理 | Pinia | 2.2.6 |
| UI 组件库 | Ant Design Vue | 4.2.6 |
| 图标库 | @ant-design/icons-vue | 7.0.1 |
| 运行时 | Node.js | 18.17.0 |
| 包管理器 | npm | 9.6.7 |

## 四、代码质量评估

### 4.1 架构优势

✅ **状态管理清晰**：Pinia 四个 Store 职责分明，依赖关系简单  
✅ **组件化良好**：组件拆分合理，单一职责原则  
✅ **虚拟滚动实现正确**：HexEditor 使用 `transform: translateY()` 优化大文件性能  
✅ **插件系统可扩展**：BaseInterpreter 抽象设计良好，支持用户自定义  
✅ **Piece Table 设计**：为高效编辑大文件预留了正确架构

## 八、关键技术决策记录

### 8.1 为何使用 Piece Table？
**决策**：使用 Piece Table 而非 Gap Buffer  
**理由**：
- 插入/删除操作 O(1) 复杂度（仅修改片段表）
- 支持高效撤销/重做（记录片段变化）
- 内存占用小（不移动原始数据）
- VS Code 等编辑器的成熟方案

**当前状态**：框架完成，核心算法未实现

### 8.2 为何使用虚拟滚动？
**决策**：仅渲染可见行  
**理由**：
- 100MB 文件约 600 万行，全渲染会导致 DOM 节点过多
- 虚拟滚动可将渲染节点控制在 50-100 个

**当前实现**：
- `HexEditor.vue` 计算 `startRowIndex` 和 `visibleCount`
- `transform: translateY()` 偏移虚拟行

**优化空间**：
- 动态调整 `OVERSCAN`（根据滚动速度）
- 缓存行解码结果（UTF-8 解码耗时）

### 8.3 为何使用插件化解释器？
**决策**：BaseInterpreter 抽象 + PluginManager 注册  
**理由**：
- 用户可自定义数据解释逻辑（如私有协议字段）
- 解释器可独立测试
- 支持异步计算（Web Worker）

**扩展性**：
- 用户可在设置中加载外部 JS 脚本
- 解释器可发布为 npm 包

### 8.4 为何使用 .TAG 文件？
**决策**：标签信息存储在独立文件而非文件元数据  
**理由**：
- Web 环境无法修改原始文件的扩展属性
- JSON 格式易于版本控制（Git）
- 可跨平台共享标签（Windows/Linux/macOS）

**缺点**：
- 用户需手动管理 .TAG 文件
- Electron 版本可改进为自动加载
